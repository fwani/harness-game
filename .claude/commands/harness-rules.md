> 덮어쓰지 않는다(.suggested로만 보존). 프롬프트만 고치려면 이 파일을 직접 편집한다.

너는 이 레포의 **변경 가능한 에이전트 룰**을 사용자와의 질문으로 설정한다.
최종 산출물은 `.agent-harness.yml`의 일부 섹션을 갱신하는 것이다. 추측하지 말고,
모르면 질문하고, 보안은 약화시키지 않는다.

3. 감지 정보로 **맥락에 맞는 질문**을 만든다. 예: 소스 디렉터리가 `src/domain`, `src/infra`
   라면 "domain → infra 의존을 금지할까?"처럼 실제 경로로 묻는다.

## 1. 네 가지 범위를 순서대로 인터뷰한다

각 범위마다: 감지 정보 기반 **추천안을 먼저 제시**하고, 사용자가 수정/확인하게 한다.
질문은 한 번에 하나씩, 짧게. 사용자가 "스킵"하면 그 섹션은 현재 값을 유지한다.

### A. 아키텍처 경계 → `architecture`
- `layers`: 레이어 이름 목록 (예: `domain`, `application`, `infrastructure`).
- `forbidden_imports`: `{ from, pattern }` 목록. `from` 경로의 파일이 `pattern` 문자열을
  import하면 `agent-harness verify`가 위반으로 잡는다.
  예: `{ from: "src/domain", pattern: "infrastructure" }`.
- `dependency_direction`: 허용된 의존 방향 (예: `infrastructure -> domain`).

### B. 보안 패턴 → `security` (추가만 가능, 제거 금지 — 아래 3절)
- `secret_patterns`: 커밋 금지 파일 글롭 (예: `.env`, `*.pem`, `*.key`).
- `sensitive_log_terms`: 로그에 남기면 안 되는 단어.
- `destructive_commands`: 차단할 파괴적 명령 정규식. `agent-harness guard`가 사용한다.

### C. 에이전트 작업원칙 → `agent` (보호 키 제외)
토글 가능: `no_guessing`, `minimal_change_scope`, `reuse_before_create`,
`concise_output`, `require_doc_spec_sync`, `require_tests_for_code_changes`,
`require_docs_for_behavior_changes`, `working_loop_required`.
**`prohibit_destructive_commands_without_confirm`은 보호 키 — 끄지 않는다.**

### D. 에스컬레이션 항목 → `agent.require_human_escalation_for`
사람 확인이 필요한 상황 목록. **추가만 가능**, 기본/조직 정책 항목은 제거할 수 없다.

## 2. 결과를 `.agent-harness.yml`에 병합 저장

- 해당 섹션(`architecture`, `security`, `agent`)만 갱신하고 나머지는 그대로 둔다.
- `version`, `project`, `commands`, `docs`는 건드리지 않는다.
- YAML 형식을 깨지 않게 저장한다.

## 3. 보안 바닥(security floor) — 절대 약화 금지

- `agent.prohibit_destructive_commands_without_confirm`는 항상 `true`.
- `security.destructive_commands`는 기존 항목을 **제거하지 않는다**(추가만).
- `security.secret_patterns`는 비울 수 없다.
- `agent.require_human_escalation_for`는 기존 항목을 **제거하지 않는다**(추가만).
- 의심되면 약화하지 말고 사용자에게 확인한다.

## 4. 검증으로 끝낸다

저장 후 반드시 실행한다:

```bash
agent-harness verify
```

`verify`는 보안 바닥 위반을 기계적으로 잡는다. 실패하면 약화된 키를 되돌리고
다시 검증한다. **검증을 실제로 통과하기 전에는 "완료"라고 보고하지 않는다.**

마지막에 변경한 섹션·추가한 항목·verify 결과를 간결하게 요약한다.
