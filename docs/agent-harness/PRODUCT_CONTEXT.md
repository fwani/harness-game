# Product Context

> Generated by company-agent-harness v0.4.0. 온보딩 중에 사람이 채운다.

## What is this?

플래시 게임 레파지토리임. 카드게임, 홀짝게임, 오목, 바둑, 장기 등 간단하게 게임을 할 수 있고, 멀티로 진행도 되며, 기록을 할 수 있는 게임시스템임.

> Status가 비어 있으면 한 문단으로 제품 목적을 적는다.

## Primary users

> 제품 설명과 현재 웹 UI(`src/ui`)에서 도출. 정식 사용자 리서치는 아직 없음.

- 브라우저에서 가볍게 한 판 즐기려는 **캐주얼 플레이어**. 빠른 진입·간단한 조작,
  명확한 승패 피드백이 목표다.
- 이 레포로 에이전트 자동개발을 운영/검증하는 **개발자·운영자**.

## Key flows

> 현재 웹 UI에서 관찰한 핵심 흐름. UI/UX 세부·갭은
> `docs/agent-harness/UX_GUIDELINES.md` 참고.

1. **게임 선택 → 플레이 → 결과 확인**: 상단 탭에서 게임을 고르고, 조작 후
   차례/점수/승패 결과를 화면에서 확인한다(예: 가위바위보, 홀짝, 오목).
2. **보드 게임 한 판**: 2인 로컬로 수를 두고 턴이 전환되며 종료/승자가 표시된다
   (오목은 승자 판정까지, 바둑·장기는 일부만 — UX_GUIDELINES의 "현재 UI 상태" 표).
3. **카드 딜**: 인원·장수를 입력해 덱을 섞어 나눠준다.

## UI/UX

> **게임 제품이므로 UI/UX는 도메인 규칙과 동급의 제품 요소다.** 새 기능·이슈를
> 만들 때(이슈 생성기 포함) UI/UX 측면을 함께 점검한다. 원칙·체크리스트·현재 UI
> 상태·알려진 갭은 **`docs/agent-harness/UX_GUIDELINES.md`** 에 정리돼 있다.
>
> 핵심: 도메인/애플리케이션 로직이 구현돼도 플레이어가 화면에서 쓸 수 없으면
> 미완성이다(예: 장기 이동 로직은 있으나 UI는 보기 전용). 규칙 이슈를 만들면
> 그것을 플레이어가 실제로 쓰는 **UI 연동 이슈가 짝으로 필요한지** 확인한다.
>
> 게임별 규칙·UI 요구사항·구현 상태(✅/⚠️/❌)는 **`docs/games/`** 에 게임 단위로
> 정리돼 있다. 새 게임/규칙/UI 이슈를 만들 때 해당 게임 사양을 함께 갱신한다.
