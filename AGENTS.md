## Mission

이 문서는 AI 에이전트가 이 레포지토리에서 안전하고 일관되게 작업하기 위한 시작점이다.
목표는 에이전트가 **덜 추측하고, 더 검증하고, 더 일관되게** 일하도록 만드는 것이다.

## Working agreement (작업 원칙)

> "통제된 자율(controlled autonomy)": 권한은 받되, 결과는 의심하고 검증한다.

- **추측하지 않는다.** 코드·문서·실제 실행 결과에 근거해 판단한다. 모르면 찾아보거나 묻고, 모르는 값은 채우지 않는다.
- **최소 변경.** 요청한 범위만 손댄다. 시키지 않은 리팩토링·파일 생성·구조 변경은 하지 않는다.
- **중복을 만들지 않는다.** 새로 만들기 전에 기존 함수·유틸·패턴을 먼저 찾아 재사용한다.
- **검증으로 끝낸다.** "변경했다"로 멈추지 말고 실제 반영·동작을 확인한 뒤 보고한다.
- **문서와 동기화한다.** 동작/API/설정/운영이 바뀌면 같은 작업에서 docs·spec·tasks 반영을 점검한다.
- **간결하게.** 군더더기·상투적 재요약 없이 무엇을/왜/어떻게만 전달한다.
- **파괴적 명령은 사람 확인 전 실행 금지.** 로컬 변경이나 데이터를 잃는 명령은 멈추고 확인받는다(아래 Security rules).

## Standard agent working loop

1. 요청을 한 문장으로 재정리하고 **범위(건드릴 것/건드리지 않을 것)** 를 명시한다.
2. 관련 문서·코드를 확인하고, **재사용 가능한 기존 함수·유틸·패턴을 먼저 찾는다.**
3. 작은 실행 계획을 작성한다(비자명한 작업은 계획 먼저).
4. 최소 범위로 변경한다.
5. 테스트, 린트, 타입 체크, 빌드를 실행한다 (`scripts/agent-harness/agent-verify`).
6. 실패하면 원인을 분석하고 수정한다.
7. **변경이 실제로 반영·동작하는지 확인한다** ("바꿨다"는 보고를 그대로 믿지 않는다).
8. 동작/API/설정/운영이 바뀌면 docs·spec·tasks 반영을 점검한다.
9. 변경 요약, 검증 결과, 남은 위험을 간결하게 보고한다.

## Quality rules

- 테스트 없는 기능 추가를 피한다.
- 버그 수정에는 가능한 한 회귀 테스트를 먼저 추가한다.
- 공개 API 변경 시 문서를 업데이트한다.
- 설정, 실행 방식, 운영 방식 변경 시 관련 문서를 업데이트한다.
- 검증을 실제로 실행하지 않았다면 "통과했다"고 말하지 않는다.
- 실패한 검증 결과를 숨기지 않는다.

## Security rules

- 비밀정보, 토큰, 키를 커밋하지 않는다.
- 민감정보를 로그에 남기지 않는다.
- 인증/인가 흐름을 우회하지 않는다.
- 보안 판단이 필요한 경우 사람에게 에스컬레이션한다.
- **로컬 변경/데이터를 잃는 파괴적 명령은 사람 확인 전에 실행하지 않는다**:
  `git reset --hard`, `git checkout -- .`, `git clean -f`, `git push --force`, `git branch -D`,
  외부 리소스가 남았는데 DB만 지우는 등 정합성 깨는 삭제. 목록은 `.agent-harness.yml`의
  `security.destructive_commands`에 있고, `agent-harness guard`로 기계적으로 검사할 수 있다(docs/agent-harness/SECURITY.md).

## Architecture rules

- 아키텍처 규칙은 `ARCHITECTURE.md`를 따른다.
- 불확실한 의존성 방향은 임의로 추가하지 않는다.
- 레이어 경계가 명확하지 않으면 먼저 문서화하고 사람에게 확인한다.

## Escalate to human when

- 요구사항이 충돌한다.
- 데이터 삭제 또는 마이그레이션 위험이 있다.
- 인증/인가 정책이 바뀐다.
- 외부 고객에게 영향을 줄 수 있다.
- 비용이나 인프라 영향이 크다.
- 테스트 결과와 요구사항이 모순된다.
